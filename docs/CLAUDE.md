# CLAUDE.md

## 一、高层行为指南（行为准则）

为了减少常见的 LLM 编码错误，并提升团队协作效率，本指南提供行为准则。所有项目特定规范请参见下文。

**权衡原则**：本指南偏向谨慎优先而非快速完成任务。在简单任务中可根据实际情况灵活处理。

### 1. 编码前先思考

**不要假设，不要隐藏困惑，明确权衡。**

在开始实现之前：

- 明确陈述你的假设。如果不确定，请提出问题。
- 如果存在多种可能的理解，列出它们——不要默默选择。
- 如果存在更简单的方案，请说明。必要时提出异议。
- 如果某些地方不清楚，请停止并明确指出问题所在。

### 2. 简单优先

**写最少量代码解决问题，不做无意义扩展。**

- 不要添加未被要求的功能。
- 不要为一次性使用的代码设计抽象。
- 不要添加未被要求的灵活性或配置能力。
- 不要为不可能发生的情况添加异常处理。
- 如果原本 200 行代码可以用 50 行完成，请重写。

自问：“高级工程师会认为这过于复杂吗？”如果是，请简化。

### 3. 精准修改

**只改必要部分，清理自己引入的问题。**

编辑现有代码时：

- 不要“改进”相邻代码、注释或格式。
- 不要重构没有问题的代码。
- 保持现有风格，即使你可能会写得不同。
- 如果发现无关的死代码，记录而不是删除。

当你的修改产生孤立部分：

- 删除你引入但未使用的 import/变量/函数。
- 不要删除已有的死代码，除非被明确要求。

检验标准：每一行修改都应直接对应用户请求。

### 4. 目标驱动执行

**明确成功标准，循环验证直到确认成功。**

将任务转化为可验证目标：

- “添加校验” → 先写测试覆盖无效输入，再让测试通过。
- “修复 Bug” → 写能复现 Bug 的测试，再修复并让测试通过。
- “重构 X” → 确保测试在前后均通过。

多步任务，简要计划：

```

1. [步骤] → 验证: [检查点]
2. [步骤] → 验证: [检查点]
3. [步骤] → 验证: [检查点]

```

强成功标准可让 AI 或开发者独立循环执行。弱标准（“让它能用”）需要频繁澄清。

### 5. 注释要求

**所有代码注释必须使用中文，包括类、方法、字段、条件分支、返回值等。**

- 保证方法、类、字段和决策逻辑可读性。
- 阈值、边界条件和输入输出必须明确。
- 注释可直接转换为 docstring 或表格规范，保证 AI 助手与团队理解一致。

## 二、开发治理规范（项目特定规范）

### 1. 项目简介

本文件为 Claude Code / Codex / Gemini CLI / Copilot 提供统一工程上下文。

### 2. 目录结构说明

- `app/`：主应用代码（API、核心路由、Prompt 检索与推理逻辑）
- `config/`：运行配置（provider / model / profile）
- `prompts/`：静态 Prompt 文本注册目录
- `mcp_servers/`：本地 stdio MCP 服务端实现
- `scripts/`：运维、预处理与自动化脚本
- `tests/`：单元、集成、回归与 live 测试
- `docs/iterations/`：迭代设计与交付记录

### 3. Python 开发规范

1. 依赖管理统一使用 `uv`，新增依赖必须更新 `pyproject.toml`。
2. HTTP 调用优先使用 `httpx`，统一 timeout 与错误处理。
3. 所有新增函数/方法必须包含强类型声明（参数与返回值）。
4. 结构化数据优先使用 `dataclass` 或 `pydantic` 建模，避免裸字典扩散。
5. 关键流程必须补充中文注释，说明输入、输出和异常策略。

### 3a. 代码注释规范（表格版）

| 类/方法范围       | 注释内容维度      | 描述与要求                                                                             | 示例/阈值说明                                                                                               |
| ----------------- | ----------------- | -------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------- |
| 所有类            | 类功能说明        | 描述类的职责、核心输入输出、核心能力、设计目标                                         | `PlannerPolicy`：动态生成 routing execution plan，adaptive & risk-aware                                     |
| 类方法/静态方法   | 方法功能说明      | 描述方法的总体作用，输入、输出及副作用                                                 | `evaluate`：分析 query 与 memory，返回 PlannerDecision                                                      |
| 方法参数          | 参数作用          | 说明每个参数在逻辑中的具体作用和影响                                                   | `query`: 用户输入文本，用于复杂度、风险判断；`memory_hits`: 历史执行记录，用于判断 rewrite 是否有价值       |
| 方法字段/局部变量 | 字段作用与逻辑    | 每个字段如何参与决策、条件判断或影响返回值                                             | `token_count`：单词数，>14 判定复杂; `fallback_rate`：历史 fallback 占比，高于0.5 → 启用 rewrite            |

| 条件分支          | 判定逻辑说明      | 描述触发条件、判断顺序、优先级，以及每个条件对应的结果                                 | 高风险 keyword → query_complexity = "high_risk" → risk_level = "high"                                       |

| Execution Plan    | Pipeline 构建说明 | 说明不同 risk_level / should_rewrite 状态下，pipeline 构建规则及各 step 位置           | 高风险：["classify","fallback","llm_answer","memory_write","trace_write"]<br>正常风险：rewrite 插入 index 0 |
| Planner Reason    | 决策原因构建      | 列出生成 planner_reason 的组成部分，包含复杂度、风险、memory 决策、触发信号等          | "complexity=complex;risk=medium;memory=memory_rewrite_success"                                              |
| 返回值            | 输出结构说明      | 说明方法返回值类型、各字段含义及与决策逻辑的关系                                       | `PlannerDecision`: execution_plan, query_complexity, risk_level, planner_reason                             |
| 注释风格          | 可读性与完整性    | 注释必须能让人快速理解方法输入、输出、字段作用、决策逻辑、阈值和边界条件；必须使用中文 | 参考 `PlannerPolicy.evaluate` 注释完整表述                                                                  |

### 使用要求

1. 新增或修改类/方法时，AI 助手必须生成对应表格版注释内容，并确保使用中文。
2. 每个字段、条件、返回值必须覆盖到表格维度。
3. 阈值和边界条件必须明确。
4. 注释必须清晰描述逻辑流、优先级和触发条件。
5. 表格内容可直接转换为方法或类的 docstring，保持可读性和可操作性。

### 4. MCP 优先原则

1. Agent 工具调用优先走 MCP 协议，不直接 import 工具函数。
2. MCP server 统一采用 stdio transport，不引入 HTTP transport。
3. 工具返回值应结构化（JSON 可序列化），便于审计与编排。

### 5. Prompt 管理规范（MCP prompt_registry）

1. Prompt 统一存放 `prompts/*.txt`。
2. 读取/更新统一通过 `prompt_registry` MCP 工具执行。
3. 禁止在业务代码中新增硬编码系统 Prompt；如需新增必须同步落盘。

### 6. LangGraph 规范

1. 节点职责单一，节点间通过显式状态对象传递数据。
2. 路由分支条件需可观测（日志可追踪判定依据）。
3. 回退策略（fallback）必须可测试，并具备阈值配置入口。

### 7. 测试规范

1. 新增功能至少提供对应单元测试；跨模块行为补充集成测试。
2. 测试命名需体现行为意图，避免抽象命名。
3. 修改配置与脚本时至少执行相关 smoke check。
4. 对外部网络依赖应支持 mock 或 graceful degrade，保证 CI 稳定。

### 8. 文档与治理强制同步规则

1. **任何行为变更**（路由结果、阈值、策略、默认值）必须同时更新 `README.md` 与 `docs/project_spec.md`。
2. **任何 graph/node/edge/state 变更**必须同步更新 README 中 Mermaid 图与 `docs/project_spec.md` 的状态/节点说明。
3. **新增脚本或脚本行为变化**必须更新 README 的 scripts commands。
4. **新增依赖**（测试/脚本/示例所需）必须更新 `pyproject.toml`。
5. **API schema 变化**必须更新 README 中的 curl 与响应示例。
6. **新增功能**必须补齐对应测试（至少 unit，涉及跨模块需 integration/regression/api；真实依赖路径需 live）。

### 9. Interview Knowledge Base 同步规则

1. 任何重要架构、策略、执行流程、fallback、memory、planner、runtime 变更，若可能影响面试表达，必须同步更新 `docs/interview_questions.md`。
2. 新增核心模块时，必须新增对应 interview question。
3. 删除或重构核心模块时，必须同步更新或删除相关 interview question。
4. `interview_questions.md` 必须仅记录“真实实现过”的内容，不允许虚构设计。

## 十、AI 助手代码注释使用说明（中文模板）

**目的**：确保 AI 助手在修改或新增任何代码时，遵循高层行为准则，并生成符合项目治理规范的中文注释。

### 1. 基本行为要求

1. **先思考再编码**
   - 明确任务假设与边界条件。
   - 如果多种实现方式，列出选择理由。
   - 不理解或模糊的地方，先提出问题或注释。

2. **简单优先**
   - 只实现任务要求的功能，避免冗余。
   - 一次性使用的逻辑无需额外抽象或配置。
   - 保持代码精简可读，逻辑清晰。

3. **精准修改**
   - 仅修改必要代码，不调整无关代码或注释。
   - 删除自己新增但未使用的 import/变量/函数。
   - 保持原有风格和逻辑，避免随意重构。

4. **目标驱动执行**
   - 将每个任务拆成可验证步骤。
   - 对每一步提供验证标准（如单元测试或逻辑检查）。

### 2. 注释生成模板

> **适用于类 / 方法 / 函数 / 字段 / 条件分支 / 返回值等**
> **必须使用中文**

#### 类注释模板

```python
class 类名:
    """
    类功能：描述类职责、核心输入输出、核心能力与设计目标
    使用场景：说明类适用的业务场景或模块
    """
```

#### 方法/函数注释模板

```python
def 方法名(参数1, 参数2):
    """
    方法功能：说明方法作用及核心逻辑
    参数说明：
        参数1：作用与影响
        参数2：作用与影响
    返回值：说明返回数据类型与字段意义
    异常处理：说明方法可能抛出的异常及触发条件
    """
```

#### 字段 / 局部变量注释模板

```python
字段名 = 初始值  # 说明字段作用、逻辑参与情况、阈值和边界条件
```

#### 条件分支注释模板

```python
if 条件:
    # 说明触发条件、判断顺序、优先级及对应处理结果
```

#### 决策结果与执行计划注释模板

```python
# 输出影响：
#   should_rewrite = True → rewrite 插入到 execution_plan index 0
#   planner_reason = "complexity=complex;risk=medium;memory=memory_rewrite_success"
```

### 3. 使用流程示例（AI 助手操作）

1. 阅读任务指令，明确目标、边界与假设。
2. 判断是否为新增代码或修改代码：
   - 新增 → 使用类/方法/字段注释模板
   - 修改 → 保持原有风格，更新必要注释

3. 对每个字段、条件分支和返回值生成中文注释，覆盖表格维度。
4. 生成 docstring 并嵌入到方法/类中。
5. 验证注释与逻辑一致性，确保可直接执行或转换为表格说明。
